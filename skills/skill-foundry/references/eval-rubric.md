# スキル完成度ルーブリック（10軸）

スキル（SKILL.md）そのものの**完成度**を測る判断指標。
Worker 成果物の合否は各スキルの出力契約＋自己検証で判定する（本ルーブリックの対象外）。

> ⚪ 配点・合格ラインは初期値。運用で調整する。

## 位置づけ

- **評価対象:** スキル一式（SKILL.md + references/）
- **評価者:** Opus（メインセッションまたは Opus サブエージェント）。スキルを書いた実行者自身の採点は参考値とし、配備判定は別の実行者が行う
- **タイミング:** ①クリエイト完了時（配備可否の判定） ②アップデート後（前後スコア比較） ③定期棚卸し（DEBT_INVENTORY と連動）

## 10軸（各 0 / 1 / 2 点）

| # | 軸 | 2点の基準 | 0点の例 |
| --- | --- | --- | --- |
| 1 | 起動精度 | description だけで「何を+いつ」が一意にわかり、他スキルと競合しない | いつ呼ばれるか読み取れない・複数スキルと重複 |
| 2 | 形式適合 | validate PASS、命名・ディレクトリ構成が規約どおり | validate FAIL |
| 3 | 実行者適合 | 想定実行者を明記し、プロファイル（Manager=判断基準/Worker=手順列挙）に合った書き分け | 実行者不明、判断と手順が混在 |
| 4 | 入出力契約 | 入力・出力（Manager は委任内容と受け入れ基準）が仕様として曖昧なく定義 | 何を渡し何が返るか読み手が推測する必要がある |
| 5 | 手順の実行可能性 | 想定モデルが行間を埋めずに実行できる粒度 | Sonnet 5 に裁量判断を要求するステップが残る |
| 6 | 自己完結性 | コールドスタートで成立（前提・パス・規約が本文内で完結） | 会話文脈や暗黙知がないと実行不能 |
| 7 | 接地 | 断定に出典と根拠強度（🟢🟡⚪）、sources.md に分離 | 出典なしの断定・数値 |
| 8 | 検証と完了条件 | 機械的チェックが定義され、全段階に測定可能な終了基準 | 完了条件がない・「良くなったら終了」等の主観基準 |
| 9 | 簡潔性 | 本文500行以内、progressive disclosure、削れる行がない | 背景知識が本文に堆積・重複記述 |
| 10 | 保守性 | 失敗例セクションが運用で更新され、分割トリガーを監視できる | 失敗例が空のまま・肥大化しても分割基準がない |

## 判定（⚪初期値）

- **配備可:** 合計 16 点以上 かつ 0点の軸がない
- **要改善:** 合計 12〜15 点 → 低スコア軸を指摘して改良後に再採点
- **配備不可:** 合計 11 点以下 または 0点の軸がある → skill-foundry ②から作り直し

## 記録

- 採点はコミット・PR 本文に1行で残す（例: `rubric: 2,2,2,1,2,2,2,2,1,1 = 17 PASS`）
- アップデート（③）では前後スコアを比較し、下がった軸がある変更は採用しない
- 低スコアが常態化した軸は、テンプレート側（skill-template-*.md）の改善課題として還元する
