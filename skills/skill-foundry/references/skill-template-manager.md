# Manager スキルテンプレート（実行者: メインセッション / Opus）

判断が支配的な業務用。原則と判断基準を示し、裁量の余地を**意図的に残す**。
手順の完全列挙が必要な作業が混ざったら、その部分は Worker スキルに切り出す。

```markdown
---
name: {{skill-name}}
description: {{何をするスキルか}}。{{どんな依頼・状況のときに使用するか}}。
---

# {{Skill Title}}

> 想定実行者: メインセッション（Opus / Fable 5）または Opus サブエージェント（Manager プロファイル）

## 役割（Role）
{{担う専門的立場と、責任範囲（何を判断し、何を委任するか）}}

## 目的（Objective）
{{達成すべきゴール。測定可能に}}

## 判断基準（Judgment Criteria）
- {{この業務の中核判断と、判断の拠り所（原則・トレードオフの優先順位）}}
- {{迷ったときのデフォルト（例: 「標準機能で可能なら自作しない」）}}

## 制約（Constraints）
- {{守るべきルール。肯定文で}}
- 実装・仕組みの新設前に「Claude Code 標準機能で可能か」を確認する（[model-routing.md](model-routing.md)）

- 成果物の受け入れは、委任先スキルの出力契約への適合と自己検証の PASS で判定する

## 手順（Methodology）
1. {{分解 → 委任 → 統合 → 検証 の流れを3〜5段階で}}
2. {{各段階の成果物と、次段階へ進む判断条件}}

## 失敗例と対策（Pitfalls）
- **{{事象}}:** {{原因}} → **対策:** {{対策}}

## 完了条件（Completion Criteria）
{{終了の判断基準。受け入れ採点の合格ラインを含める}}

## 出典
[`references/sources.md`](references/sources.md) を参照。
```

## 記入時の注意
- Worker と違い、手順の網羅より**判断基準の明文化**を優先する（Opus は行間を埋められる）
- 「委任」セクションを持たない Manager スキルは設計を疑う（自分で全部やるなら Worker 分割が未了）
