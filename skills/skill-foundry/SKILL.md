# Skill Foundry — スキル鍛造パイプライン

専門エージェント（AI Company の各役職）を支えるスキルを、
**リサーチ→クリエイト→アップデート→分割** の反復で世界水準に育てるための手順書。

## 役割（Role）
スキル鍛造の**オーケストレーター**。自らは判断・統合・検証を担い、単純作業はサブエージェントに委任する。

## 目的（Objective)
対象ドメインの一流の思想・方法論を、検証可能な出典付きで `<name>/SKILL.md` に凝縮し、
起動精度（description）と実行品質（本文）を反復改善する。

## 制約（Constraints）
- スキル形式はリポジトリ README の規約に**必ず**従う（`<name>/SKILL.md`、frontmatter に `name`/`description`、`name` は小文字・数字・ハイフンのみで予約語 `anthropic`/`claude` 禁止）
- 仕組みの新設前に「Claude Code 標準機能で可能か」を必ず確認し、標準機構（skills / agents / hooks / permissions）で実現できるものを自作しない（[references/model-routing.md](references/model-routing.md)）
- スキルは実行者プロファイル（Manager = Opus / Worker = Sonnet 5）で書き分ける。モデル紐付けはスキルではなくエージェント定義・呼び出し側が持つ
- すべての定量的・断定的主張に根拠の強さ（🟢一次 / 🟡二次一致 / ⚪目安）を付し、`references/sources.md` に出典を分離する
- 本文は短く保つ（目安 500 行以内）。超えたら第④段階「分割」を発動する
- 否定文より肯定文で書く。各段階に完了条件を明示し、無限改良ループを防ぐ
- モデル本体の能力を誇張しない（インコンテキストの工夫は「引き出す」のであって「別モデル化」ではない）

## ツール（Tools）とモデルルーティング

| 作業 | 担当 | 理由 |
| --- | --- | --- |
| 方針決定・知識の統合・最終編集 | メインセッション（Fable 5 / Opus） | 判断の質が支配的 |
| 出典収集・要約・リンク検証・表整形などの単純作業 | **Sonnet 5 サブエージェント**（`Agent` tool, `model: sonnet`） | 速く安価。メインの文脈も汚さない |
| 実装後の独立レビュー | **Opus サブエージェント**（`model: opus`） | 利害のない第三者検証 |
| 形式チェック | `scripts/validate_skill.py` | 機械検証は人手より確実 |

サブエージェントへの指示は必ず「役割・目的・制約・完了条件」を含め、**結論だけ**を返させる。
Worker 成果物の受け入れは、そのスキルの出力契約への適合と自己検証の PASS で判定する。

## パイプライン

### ① リサーチ（Research）
対象ドメインの一流の思想を収集する。
1. 対象エージェント（`references/agent-roster.md` 参照）の**判断業務**を3〜5個に言語化する
2. Sonnet 5 サブエージェントを並列起動し、各判断業務ごとに「第一人者・古典・現行ベストプラクティス・アンチパターン」を WebSearch で収集させる（出典URL必須、各テーマ結論200字以内）
3. メインで統合し、採用/不採用を判定。採用分に 🟢🟡⚪ を付す
- **完了条件:** 出典付きの知見が判断業務ごとに3件以上そろい、矛盾が解消されたら終了

### ② クリエイト（Create）
1. `references/skill-template.md` でプロファイル（Manager / Worker）を選び、対応テンプレートを複製して本文を構成する
2. `description` には「何をするか」と「いつ使うか」の両方を書く（起動判定はここに依存する）
3. 出典を `references/sources.md` に分離する
4. `python3 skills/skill-foundry/scripts/validate_skill.py <skill-dir>`（リポジトリ直下から） で形式を機械検証する
- **完了条件:** validate が PASS し、想定ユーザー発話3つに対して description が起動を正しく説明でき、[references/eval-rubric.md](references/eval-rubric.md) の完成度採点で「配備可」になったら終了

### ③ アップデート（Update）
運用で得た失敗・新知見を反映する。
1. 失敗事例を「事象→原因→対策」で本文の失敗例セクションに追記する
2. 陳腐化した出典・数値を Sonnet 5 サブエージェントに再検証させ、変わったものを差し替える
3. 変更前後で同一タスクを実行し、出力差分を観測してから採用する（検証なき「改善」は採用しない）
- **完了条件:** 差分検証で改善または同等を確認し、validate が PASS し、完成度スコア（eval-rubric）が変更前を下回っていなければ終了

### ④ プロフェッショナル化 = 分割（Professionalize / Split）
スキルが育って肥大化したら、専門特化スキルへ分割する。
**分割トリガー（いずれか該当で発動）:**
- 本文が 500 行を超えた
- description の「いつ使うか」が2ドメイン以上にまたがり、起動が曖昧になった
- 特定セクションだけが更新頻度・参照頻度で突出している

**分割手順:**
1. 突出セクションを新スキル `<domain>-<specialty>/SKILL.md` として独立させる
2. 詳細知識は `references/` に逃がし、SKILL.md 本文は判断手順だけ残す（progressive disclosure）
3. 元スキルには新スキルへの参照を1行残し、重複記述を削除する
4. 両方を validate し、Opus サブエージェントで「分割前にできたことが分割後もできるか」をレビューする
- **完了条件:** 新旧両スキルが validate PASS + Opus レビュー PASS で終了

## 品質ゲート（全段階共通）
1. **形式:** `validate_skill.py` PASS
2. **接地:** 断定はすべて出典付き。目安は⚪明記
3. **起動:** description だけ読んで「いつ呼ばれるか」が一意にわかる
4. **簡潔:** 削れる行がないか毎回1周剪定する
5. **検証:** 変更はビフォー/アフター比較か第三者レビューを経て採用する

## 完了条件（Completion Criteria）
依頼された段階（①〜④）の完了条件を満たし、品質ゲート5項目を通過し、
変更をコミットしたら本スキルの1サイクルは終了。次サイクルは新たな依頼で開始する。
