skills リポジトリの棚卸しを以下の順で実行せよ。各段階の結果を集めて最後に DEBT_INVENTORY.md へ追記する。

## 1. 形式検証
`python3 skills/skill-foundry/scripts/validate_skill.py skills/*` を実行し、FAIL があれば即修正する。

## 2. 独立採点
skill-scorer エージェントに採点を委任する（自分で採点しない）。**5スキルずつのバッチ**で依頼し、
バッチ完了ごとに結果をメモに残す（中断しても完了バッチから再開できるようにする）。
前回スコア（DEBT_INVENTORY.md の最新採点表）と比較し、**下がった軸があるスキルは、
その原因となった変更をこの棚卸し内でコミットしない**（改善提案として記録に留める）。

## 3. 起動精度チェック
`tests/trigger-cases.md` の正例（発話→期待スキル）と負例（どのスキルも起動すべきでない発話）を
全スキルの description と照合して ◯/△/× を判定する。
△（競合）・×（不発）・負例での誤発動は、該当スキルの description 改善案を作る。

## 4. 失敗還元
`FAILURE_LOG.md` の「未処理」行それぞれについて、skill-foundry ③アップデートの手順で
該当スキルに対策を反映し、行を「還元済(コミットID)」に更新する。

## 5. 鮮度確認
各スキルの sources.md から、時効性の高い出典（API仕様・規制・廃止予定）を3件程度抽出し、
Sonnet 5 サブエージェントに現行性の確認を委任する。変わっていた記述は skill-foundry ③で差し替える。

## 6. 記録とコミット
結果（採点表・起動精度・還元件数・鮮度差分）を DEBT_INVENTORY.md に日付付きで追記し、
変更をコミットする。コミット本文に `rubric:` 行を含めること。

制約: 破壊的変更（スキルの削除・大幅書き換え）はこのコマンド内で行わず、提案として記録に留める。

データフロー（一方向。逆流・重複記載をしない）:
`FAILURE_LOG.md`（生ログ）→ skill-foundry ③（スキルへの反映）→ `DEBT_INVENTORY.md`（構造化された記録・採点履歴）
